# Hotfix v1.10.1 - Carga diferida + guard del motor CodeGraph

Branch: `hotfix/v1.10.1-codegraph-lazy` (desde `main`). Sin merge, sin tag.

## Problema

La v1.10.0 empaquetada no abría. `electron/carto/graph-engine.ts` tenía un import
de **valor** al tope del módulo:

```ts
import { CodeGraph } from '@colbymchenry/codegraph';
```

Ese módulo lo importa `electron/ipc/carto-graph.ts`, que se registra al arranque del
proceso main (`registerCartoGraphHandlers`). Resultado: el `require` estático de
`@colbymchenry/codegraph` corría al abrir la app, cargaba tree-sitter → grammars →
`web-tree-sitter`, que en el paquete empaquetado **no resuelve** (`Cannot find module
'web-tree-sitter'`), y tumbaba TODO el main **antes** de mostrar ventana.

Las llamadas reales al motor (`isInitialized`/`open`/`init`) ya vivían dentro de
funciones, no al arranque. El único problema era el **import de valor** estático.

## Objetivo

Que la app abra SIEMPRE. Cartografía puede fallar con gracia; el resto de GitCron no
puede depender del motor para arrancar. Cero cambios en lógica de Git ni en otras
features: sólo se tocó la CARGA del motor.

## Cambios

### Tanda 1 — Import dinámico + guard (`electron/carto/graph-engine.ts`, `lib/carto-types.ts`)

- El import de **valor** `import { CodeGraph }` se eliminó del top-level. Se mantiene
  como **solo tipos**: `import type { CodeGraph, IndexProgress, Node } from '@colbymchenry/codegraph'`.
  `import type` no genera `require` en runtime, así que ya no hay carga nativa al arranque.
- Nueva carga diferida `loadCodeGraph()`: `const mod = await import('@colbymchenry/codegraph')`,
  **cacheada** en la variable de módulo `codeGraphClass` (no reimporta en cada llamada),
  envuelta en `try-catch`. Si el módulo nativo falta, lanza la clase de error tipada
  **`CartoEngineUnavailableError`** (exportada), nunca un throw crudo.
- `buildIndex` obtiene el motor con `const Engine = await loadCodeGraph()` y usa
  `Engine.isInitialized/open/init` (antes `CodeGraph.*`). Es la ÚNICA función que usaba
  la clase como valor.
- `ensureGraph` traduce `CartoEngineUnavailableError` al estado dedicado
  **`engine-unavailable`** (en vez de un `error` genérico); cualquier otro fallo sigue
  siendo `error`.
- `lib/carto-types.ts`: `CartoGraphState` suma `'engine-unavailable'`.

**Verificación de alcance:** `grep` sobre `@colbymchenry/codegraph` en todo el repo. El
único import de valor era `graph-engine.ts:28`. Los demás usos (`lib/carto-from-codegraph.ts`
y los tests) ya eran `import type`. No queda ningún import de valor del motor en el
top-level de un módulo que se cargue al arranque.

### Tanda 2 — Degradación en el renderer (i18n + componentes)

- `lib/i18n.ts` (ES/EN/ZH): nuevas claves `cartography.graph.engineUnavailable` y
  `cartography.graph.engineUnavailableHint`.
- `components/cartography/SemanticGraphLens.tsx`: rama dedicada para
  `status.state === 'engine-unavailable'` que muestra el aviso (título + detalle).
  `GraphHint` acepta un `detail` opcional.
- `components/cartography/CartoPanoramaHeader.tsx`: el panel de resumen muestra el
  aviso en `engine-unavailable` en vez de caer a "vacío".
- Test `electron/__tests__/carto-graph-engine-unavailable.test.ts`: mockea el motor para
  que su carga falle y verifica que `ensureGraph` degrada a `engine-unavailable` **sin
  propagar la excepción**.

**Dónde quedó el guard.** Por la arquitectura, el motor se carga sólo en `buildIndex`,
que corre en **background** desde `ensureGraph` (`buildIndex(...).catch(...)`). El fallo
nunca llega de forma síncrona a un handler IPC: `carto:graph-ensure` devuelve el estado
inicial `indexing`, y el estado `engine-unavailable` se entrega al renderer por el push
`carto:graph-progress` (al que `CartographyView` ya está suscrito). Los demás handlers
usan `readyGraph()`, que devuelve `null` cuando el índice no está listo, así que nunca
tocan la clase del motor. Por eso el guard vive en `ensureGraph` (main) + el manejo del
estado `engine-unavailable` en el renderer, y `carto-graph.ts` no necesitó cambios: ya
devolvía resultados estructurados y nunca propaga la excepción.

**Sobre el screenshot del checkpoint.** Verificar visualmente el aviso exige correr la
app Electron empaquetada, inviable en este entorno no interactivo. En su lugar, la ruta
degradada se cubre con el test automatizado descrito arriba (fuerza el fallo de carga y
comprueba el estado `engine-unavailable`, que es exactamente lo que dispara la rama de UI
del aviso). Para reproducirlo a mano: forzar que `import('@colbymchenry/codegraph')`
falle (o renombrar el módulo nativo) y abrir Cartografía; el resto de la app sigue andando.

### Tanda 3 — Cierre

- `package.json`: bump a `1.10.1`.
- `CHANGELOG.md`: entrada `[v1.10.1]`.

## Cómo degrada Cartografía (resumen)

1. Se entra a Cartografía → `ensureGraph` arranca `buildIndex` en background y devuelve `indexing`.
2. `buildIndex` llama `loadCodeGraph()`; si el módulo nativo falta, lanza `CartoEngineUnavailableError`.
3. El `.catch` de `ensureGraph` fija el estado `engine-unavailable` y lo empuja por
   `carto:graph-progress`.
4. El renderer (SemanticGraphLens / CartoPanoramaHeader) muestra el aviso i18n. El cliente
   Git no se ve afectado.

## NOTA para Ale

Este hotfix hace que la app **ABRA** aunque el motor de CodeGraph no esté disponible en el
`.exe`. Para que Cartografía **FUNCIONE** en el empaquetado todavía falta resolver el
empaquetado de `web-tree-sitter` / `tree-sitter-wasms`: hoistearlos a dependencia directa
y `asarUnpack` de los `.wasm`/`.node`. Eso es un paso de **build** aparte, fuera del alcance
de este hotfix. El tag `v1.10.1` y el rebuild los hacés vos tras QA.
